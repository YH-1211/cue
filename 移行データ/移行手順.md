# Cue 移行手順（Vercel → 別ドメインのサーバー）

このフォルダは、Cue を Vercel から **自分のサーバー（VPS）+ 独自ドメイン** へ引っ越すための一式です。
クロちゃんが用意した。落ち着いて上から順にやれば移せる。

---

## 0. 前提と全体像（まず読む）

Cue は「ただのHTMLサイト」ではなく、**Node.js のサーバーが常に動いている必要がある**アプリです。理由：

- ログイン（Supabase 認証 / Google OAuth）
- サーバーアクション（投稿・写真アップロード）
- プッシュ通知の送信（web-push）
- 定期実行（Cron＝通知バッチ・イベント自動収集）

なので移行先は「**Linux サーバー（VPS）**」が必要です。Xserver VPS / さくらVPS / ConoHa / AWS Lightsail / Hetzner などどれでもOK。
**Supabase（データベース）は引っ越し不要**です。今のまま使い続けます。新サーバーから同じ Supabase に繋ぐだけ。

引っ越しで動かす部品は3つ：

| 部品 | Vercelでは | 移行後 |
|---|---|---|
| 本体（Webサーバー） | Vercelが自動 | Docker か PM2 で常駐 |
| Cron（定期実行3本） | `vercel.json` | サーバーの crontab |
| 公開URL / HTTPS | Vercelが自動 | Nginx + Let's Encrypt |

やり方は **A: Docker（おすすめ）** と **B: PM2（Dockerなし）** の2通り。どちらか片方でいい。初めてなら **A** を推奨。

---

## 1. 事前準備（サーバーを借りる前にやること）

### 1-1. ドメインのDNS設定
移行先サーバーのIPアドレスに、独自ドメインを向ける。
- `A` レコード: `example.com` → サーバーのIP
- `A` レコード: `www.example.com` → サーバーのIP（www も使うなら）

反映に最大数時間かかる。先にやっておくと楽。

### 1-2. Supabase の許可URLに新ドメインを追加（超重要・忘れがち）
これをやらないと**ログインできなくなる**。
1. Supabase ダッシュボード → 対象プロジェクト → **Authentication → URL Configuration**
2. **Site URL** を `https://新ドメイン` に変更
3. **Redirect URLs** に以下を追加（Vercelのも残してOK）：
   - `https://新ドメイン/auth/callback`
   - `https://新ドメイン/**`

### 1-3. Google OAuth のリダイレクトURI（Googleログインを使っている場合）
Google Cloud Console → 認証情報 → OAuthクライアント → 「承認済みのリダイレクトURI」。
ここは **Supabase のコールバックURL**（`https://<プロジェクトRef>.supabase.co/auth/v1/callback`）なので、
ドメインを変えても **基本は変更不要**。Supabase側（1-2）だけ直せばよい。

## 2. コードとデータを移す

### 2-1. ソースコードを移行先サーバーへ
GitHub に上がっているので、サーバー上で clone するのが楽：
```bash
git clone https://github.com/YH-1211/cue.git
cd cue
```
（GitHubを使わない場合は、このプロジェクトフォルダを丸ごと `scp` でサーバーへ転送。ただし `node_modules` と `.next` は除く＝サーバー側で作り直す）

### 2-2. 環境変数ファイルを作る
このフォルダの `.env.example` を参考に、サーバー上で `.env.local`（または `.env.production`）を作成。
**今の `.env.local` の中身をそのままコピーすればOK**（Supabaseは同じものを使うので値は変わらない）。
`SUPABASE_*` `NEXT_PUBLIC_*` `VAPID_*` `CRON_SECRET` `ADMIN_EMAIL` 全部必要。

> ⚠️ `.env.local` は秘密情報。GitHubには上げない（`.gitignore` で除外済み）。サーバーへ手動でコピーする。

### 2-3. データベースは何もしない
イベントもユーザーも全部 Supabase（クラウド）にある。引っ越し対象外。そのまま繋がる。

---

## 3-A. デプロイ方式A：Docker（おすすめ）

### 3-A-1. 下準備（next.config に1行追加）
Docker用に軽量な「standalone」出力を有効化する。`next.config.ts` を開いて、`experimental` の隣に `output` を足す：
```ts
const nextConfig: NextConfig = {
  output: "standalone",            // ← この1行を追加
  experimental: {
    serverActions: { bodySizeLimit: "35mb" },
  },
};
```
（この変更は Vercel 側でも問題なく動くので、足したままでよい）

### 3-A-2. ファイルを配置
このフォルダの `Dockerfile` と `docker-compose.yml` を**プロジェクトのルート**（`package.json` と同じ場所）にコピー。

### 3-A-3. 起動
サーバーで Docker / docker compose を入れたうえで：
```bash
docker compose up -d --build
```
これで `http://サーバーIP:3000` でアプリが立ち上がる。次の「4. 公開」でドメイン＆HTTPSを付ける。

---

## 3-B. デプロイ方式B：PM2（Dockerを使わない）

Docker を入れたくない場合はこちら。Node.js 24 をサーバーに入れておく。

```bash
# Node 24 を入れる（nvm 推奨）
nvm install 24 && nvm use 24

cd cue
npm ci                 # 依存をインストール
npm run build          # 本番ビルド
npm install -g pm2     # プロセス管理ツール

# このフォルダの ecosystem.config.js をルートにコピーしてから：
pm2 start ecosystem.config.js
pm2 save
pm2 startup            # 表示されたコマンドを実行→サーバー再起動でも自動復帰
```
これで `http://サーバーIP:3000` で常駐する。

---

## 4. ドメイン公開 + HTTPS（Nginx）

このフォルダの `nginx.conf.example` を参考に設定。要点：
- `server_name` を自分のドメインに書き換え
- アプリ（`localhost:3000`）へリバースプロキシ
- **`client_max_body_size 35m;`** を入れる（写真アップロード対応。入れないと大きい画像でエラー）

HTTPS は **Let's Encrypt（無料）** が定番：
```bash
sudo apt install certbot python3-certbot-nginx
sudo certbot --nginx -d example.com -d www.example.com
```
証明書は自動更新される。

---

## 5. Cron（定期実行3本）を移す

Vercel の `vercel.json` に入っていた3本を、サーバーの crontab で再現する。
このフォルダの `crontab.txt` の中身を、ドメインと `CRON_SECRET` を書き換えてから登録：
```bash
crontab -e
# crontab.txt の中身を貼り付けて保存
```

> ⏰ **時刻の注意**: Vercel Cron は **UTC** 基準だった。サーバーのcronは**サーバーのローカル時刻**で動く。
> 同じタイミングにしたいなら、サーバーのタイムゾーンをUTCにするか、`crontab.txt` のコメント通り日本時間に読み替える。
> （元の設定：通知 0:00 UTC=朝9時/10:00 UTC=夜19時、収集 18:00 UTC=朝3時）

## 6. 動作確認チェックリスト

- [ ] `https://新ドメイン` でトップが開く
- [ ] HTTPS の鍵マークが付いている
- [ ] **ログイン/Googleログインが通る**（←許可URL設定ミスだとここで失敗）
- [ ] イベント一覧・詳細が表示される（＝Supabaseに繋がっている）
- [ ] イベント投稿＋写真アップロードができる（35MB上限）
- [ ] プッシュ通知の許可・受信ができる
- [ ] 翌日、Cronで通知/収集が動いたか（ログ確認）

---

## 7. 最後に：Vercelを止める

新サーバーで全部動くのを確認してから：
- DNSが完全に新サーバーへ向いたのを確認
- Vercel 側のプロジェクトは、しばらく残してから削除（すぐ消さない＝切り戻し用）

---

## 困ったとき

- **ログインできない** → Supabaseの Site URL / Redirect URLs（手順1-2）を再確認
- **写真がアップロードできない** → Nginxの `client_max_body_size 35m;`（手順4）
- **通知が来ない** → `VAPID_*` の3つの環境変数 / Cronのログ
- **Cronが動かない** → `CRON_SECRET` が一致しているか、時刻がUTC/ローカルどちらか

このフォルダの各ファイル（Dockerfile / docker-compose.yml / nginx.conf.example / crontab.txt / ecosystem.config.js / .env.example）にも個別のコメントを書いてある。
